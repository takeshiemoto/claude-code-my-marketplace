# React State-First Design

以下の原則に従って実装・レビューを行う。

## 1. 生の値ではなく状態で分岐する

観測値に対する場当たり的な条件分岐を書かない。本質的な関心事は意味のある状態である。

悪い例:

- `if (nodes.length === 0)`
- `if (status === 'success' && items.length > 0)`

良い例:

- `empty`, `loading`, `ready`, `error`, `submitting`, `disabled` などの状態を定義する
- その状態に対して明示的に分岐する
- 判別共用体 / タグ付き状態オブジェクト / 適切に命名された派生状態を使う

最初にやるべきことは、UIや機能が取りうる状態の集合を特定・定義すること。
分岐が増えたら立ち止まって状態空間をモデリングする。

## 2. 機械的に条件を追加しない

`if` をもう1つ追加して動作を拡張し続けない。
新しい条件が出現したら、まずモデル化すべき状態・遷移・不変条件がないか確認する。

推奨:

- 明示的な状態定義
- 遷移指向の思考
- 派生的な意味に対する単一の信頼できるソース

回避:

- 散在するbooleanフラグ
- 重複した条件
- ドメインの意味とレンダリングの詳細の混在

## 3. Effect駆動の同期よりイベント駆動の更新を優先する

`useEffect` は最終手段として扱う。

`useEffect` を追加する前に確認する:

- イベントハンドラで直接実行できないか
- レンダー中に計算すべき派生データではないか
- 外部システムとの同期か、それとも内部的な帳簿管理に過ぎないか

可能な限りイベント境界にロジックを移す:

- submitハンドラ
- clickハンドラ
- changeハンドラ
- 明示的なコマンド/アクション

`useEffect` は外部システムとの同期など、Reactのガイダンスに沿った正当な副作用にのみ使用する。

## 4. 便宜ではなく意味で命名する

名前はコンテキスト内で役割と振る舞いを明確にしなければならない。

伝えるべきこと:

- 状態の意味
- ドメインの意図
- 正確なアクション

曖昧な汎用動詞に懐疑的であること:

- `mutate`
- `revalidate`
- `handleData`
- `process`
- `manage`

汎用的な用語が使われている場合、そのライブラリやコンテキストで標準的であり、可読性を向上させているか確認する。

## 5. ロジックを読みやすくローカルに保つ

意図を明らかにする小さな単位を優先する。
抽出は責務が明確になる場合のみ行う。
状態遷移やドメインの意味を隠す抽象化を作らない。

目指すもの:

- 短いコンポーネント
- 短い関数
- 明示的な不変条件
- 最小限の分岐の深さ
- 予測可能なデータフロー

## 6. レビューチェックリスト

コード生成・レビュー時に以下を厳密にチェックする:

- 取りうる状態が明示的に定義されているか
- 名前付き状態があるべき箇所で生の値に対して分岐していないか
- booleanや条件がファイル/関数をまたいで重複していないか
- イベントハンドラや派生値で済むところに `useEffect` が追加されていないか
- 名前が実際のドメイン/アクションを正確に反映しているか
- コードがきれいに拡張できるか、ケースが増えるとif連鎖が爆発しないか
- ケースが10から100に増えても構造が破綻しないか

## 7. 実装バイアス

推奨する方向:

- 判別共用体
- 遷移が中心的な場合はreducer
- ソース状態から計算される派生状態
- イベント駆動のコマンド
- UI分岐のための明示的なビューモデル

回避する方向:

- 偶発的な条件分岐
- Effect多用のオーケストレーション
- 曖昧なヘルパー
- 隠れた状態遷移
- 制御不能な分岐の増殖

## 8. 出力時の振る舞い

実装・レビュー時:

1. まず状態モデルを特定する
2. 不明確な概念はコーディングの前にリネームする
3. 不要なEffectを除去する
4. 場当たり的な分岐を明示的な状態に集約する
5. ケースがスケールしても理解可能なコードを生成する
