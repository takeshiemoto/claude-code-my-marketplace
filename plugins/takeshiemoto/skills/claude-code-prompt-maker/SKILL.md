# Claude Code Prompt Maker

ユーザーの殴り書き・断片的な要望から、Claude Code（4.7）で自走させるためのプロンプトを整形するスキル。

## いつ発動するか

ユーザーが明示的にプロンプト作成を依頼したとき**のみ**：

- 「Claude Code向けにプロンプト作って」
- 「これをプロンプトにして」
- 「プロンプト整形して」
- 「殴り書きするからプロンプトに」

通常の会話・質問・依頼では発動しない。曖昧な依頼に対して「プロンプト化しましょうか？」と提案もしない。

## 4.7プロンプトの設計原則

このスキルが生成するプロンプトは、以下の原則に従う：

1. **宣言的に書く**：手順ではなく「どうなれば完了か」を渡す
2. **完了条件はチェックボックス形式**：4.7の自己検証と相性が良い
3. **判断ルールを事前に渡す**：曖昧時・詰まったときの動き方を契約
4. **触らない部分を明示**：4.7は文字通り解釈するため
5. **足場を書かない**：「3ステップごとに進捗」「最後にダブルチェック」等は書かない（4.7はネイティブに行う）
6. **ポジティブ例で書く**：「〜するな」を3つ以上並べない
7. **承認不要を明示**：自走させたいときは「私の承認は不要」「auto-acceptで進めて」を書く

## 標準骨格

最低限これを満たすプロンプトを生成する：

```text
# Goal       — 何が達成されるか（1行）
# Acceptance — チェックボックス形式の完了条件
# Constraints — 制約・禁止事項
# When stuck — 詰まったときの判断ルール
```

タスクが複雑なら以下を追加：

- `# Approach` — 手順の枠組み（細かい手順ではなく、フェーズ分け）
- `# Done` — 完了報告に含めるべき内容
- `# Notes` — 設計判断・参考にするパターン

## タスクタイプ別の追加要素

依頼内容を読み取り、タイプに応じて追加セクションを入れる。

### 機能追加

- `# Acceptance` を細かめに（正常系・エッジケース・エラー処理）
- `# Notes` で参考にする既存パターン

### バグ修正

- `# Bug` — 症状（1-2行）
- `# Repro` — 再現手順または失敗テスト
- `# Expected` — 本来あるべき挙動
- Acceptanceに「回帰防止テスト」「類似バグの確認」を含める
- 「仮説を1行で報告、そのまま修正に進んでくれ」を末尾に入れて自走と可視性を両立

### リファクタリング

- `# Scope` — 触る/触らないファイルを明示
- `# Invariants` — 壊してはいけないものをチェックボックスで
- 「変更前後の差分要約をくれ」を末尾に

### 探索・調査・設計判断

- `# Approach` で「OKまで実装しない」を明示（自走を意図的に止める）
- 選択肢を出させる構造にする
- 「調査だけでまず止まってくれ」を末尾に

### レビュー対応・コメント処理

- `# Judgement criteria` を必ず入れる（妥当/不要の判断軸）
- グレーゾーンのフォールバック（「保守的に妥当寄り」など）を明示
- 「無視・放置しない」をAcceptanceに入れる
- 判断保留の逃げ道（難しいものだけ確認を求める）を When stuck に

### ドキュメント編集・PR本文編集等

- `# Constraints` を厳格に（文字数・装飾・禁止語）
- 自己検証用の指示を `# Done` に入れる（文字数併記、禁止語チェックなど）
- gh等のCLIで実際に反映するところまでスコープに含めるか確認

## 整形のフロー

1. **殴り書きを読む**：ユーザーが書いた要素（やりたいこと・制約・条件）をすべて拾う
2. **タスクタイプを判定**：上記のどのタイプに近いか
3. **不足要素を補う**：標準骨格＋タイプ別要素を埋める。ユーザーが書いていない判断ルールやスコープ境界も、合理的なデフォルトで補う
4. **アンチパターンを避ける**：以下を含めない
   - 「丁寧に」「いい感じに」「適切に」「最善を尽くして」
   - 「3ステップごとに進捗を報告」「最後にダブルチェック」
   - 「先に計画を確認させて」（自走タスクで毎回書く必要なし）
   - 否定形を3つ以上連続
5. **コードブロックで提示**：そのままコピペできる形に
6. **仕込みポイントの解説を添える**：なぜそう書いたかを箇条書きで2-5項目（ユーザーが学習できるよう）

## 出力フォーマット

```text
[整形前の理解を1-2行で確認的に]

---

[コードブロックでプロンプト本体]

---

仕込みポイント：
- [ポイント1]
- [ポイント2]
- ...

[必要なら補足や確認質問を1つだけ]
```

## アンチパターン早見表（生成プロンプトに含めてはいけない表現）

| 含めない | 代わりに書く |
| --- | --- |
| 「いい感じに」「適切に」「最善を尽くして」 | 検証可能な完了条件 |
| 「丁寧に実装して」 | 受け入れ基準のチェックボックス |
| 「テストを書くか確認して」（質問形式） | 「テストを追加」（断言） |
| 「3ファイルごとに進捗を教えて」 | 何も書かない（4.7はネイティブに進捗を出す） |
| 「最後にダブルチェックして」 | 何も書かない（4.7は自己検証する） |
| 否定形を3つ以上連続 | ポジティブな完了像 + 必要なら具体的な禁止リスト |
| 「先に計画を確認させて」を毎回 | 自走タスクなら書かない（探索タスクのみ書く） |

## 例：殴り書き → 整形

**殴り書き入力例**：
> CodeRabbitAIのレビューに対応したい。妥当なら直す、不要ならリプライで返す

**整形時の判断**：

- タスクタイプ: レビュー対応 → Judgement criteria が必須
- 自走させる前提（明示なし） → 判断軸とフォールバックを契約
- 不足要素: 妥当性の判断軸、不要時のリプライ形式、判断保留時の動き → デフォルトで補う
- スコープ境界: 「CodeRabbitAI以外のコメントには触らない」を明示

このように、ユーザーが書いていない部分も、4.7プロンプトのベストプラクティスに沿って合理的に補完する。ただし、補完した判断ルールは仕込みポイントで明示し、ユーザーが調整できる余地を残す。

## 注意点

- ユーザーが既にCLAUDE.mdを整備していると仮定してよい（「CLAUDE.mdに従って自走で進めて」等の参照は許容）
- ユーザーが特定の指示（「リプライ形式で」「300字以内で」「DXという言葉禁止」等）を出している場合、それは絶対に Constraints か Acceptance に明記する
- 殴り書きに含まれていない情報（PR番号、ファイルパスなど）が必要な場合、整形プロンプト内では `[対象PR: #XXXX]` 等のプレースホルダーを残し、最後に「ここを埋めてください」と一言添える
- 仕込みポイントの説明は必ず添える（このスキルの価値はプロンプトそのものだけでなく、なぜそう書いたかの可視化にある）
