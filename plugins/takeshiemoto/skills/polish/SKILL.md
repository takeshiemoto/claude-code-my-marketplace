# polish

引数で渡された殴り書きを、社内同僚向けSlack/Chatwork用のビジネスチャット文に清書する。読み手視点で情報不足を検知した場合は、清書前に質問して補ってから仕上げる。

## 使用方法

```sh
/takeshiemoto:polish <殴り書きテキスト>
```

引数の殴り書きを入力として受け取る。引数が空の場合は「清書したい殴り書きを貼ってください」と促し、貼られるまで待つ。

## 前提（固定）

- **宛先**: 社内同僚（個人 または 少人数チーム）向けのビジネスチャット（Slack / Chatwork）
- これ以外（社外・上位者・全社アナウンス等）を想定するときは、ユーザーが明示した場合に限り調整する

## スタイル規範

以下を固定で適用する。

- 「〜です・ます」主体の敬語。固くなりすぎない自然なトーン
- 結論ファースト（依頼・質問・要点を冒頭に置く）
- 二重敬語・過剰表現を避ける（例: 「ご確認いただけますでしょうか」→「ご確認お願いします」）
- **「お疲れ様です」は使用禁止**（冒頭の挨拶としても、途中でも書かない）
- 「取り急ぎ」「よろしくお願いいたします」等の定型句も必要最小限に留める
- 絵文字は使用しない
- 情報量が多い場合は短い見出しや箇条書きで視認性を上げる
- 社内略称・固有名詞は初出で短く補足（読み手が知らない可能性がある場合のみ）
- 冗長な前置き・自己言及（「まとめると」「以下が清書です」等）は書かない

## 実行フロー

### Step 1: 読み手視点チェック

殴り書きを読み、以下の観点で不足がないか確認する。

- **目的種別**: 報告 / 依頼 / 相談 / 確認・質問 / 共有 / お詫び・謝意 のどれか
- **依頼系の場合**: 「何を・いつまでに・なぜ」が揃っているか
- **質問系の場合**: 何に対する質問か、回答に必要な前提が揃っているか
- **報告系の場合**: 結論と、必要であれば次アクション（誰が何をするか）が書かれているか
- **前提共有**: 読み手が知らない固有名詞・背景があるなら補足が要るか
- **期限・スケジュール**: 時間軸の明示が必要な内容か

不足があれば Step 2 へ、なければ Step 3 に進む。

### Step 2: 情報不足の補完（不足時のみ）

`AskUserQuestion` を使い、不足項目を選択肢 + 自由記述で埋めてもらう。

- 1 回の `AskUserQuestion` で最大 4 項目まで。項目が多ければ優先度の高いものから絞る
- 質問は「この情報があると読み手が判断しやすくなります」という観点で選ぶ（書き手の都合ではなく読み手視点）
- 回答が揃ったら Step 3 へ

### Step 3: トーン軸の選択

`AskUserQuestion` で文章の目的軸を 1 つ選ばせる。選択肢は以下を基本とする（殴り書きの内容に応じて文言を微調整してよい）。

- **共有・報告**: 淡々と事実と結論を伝える
- **依頼・相談**: 相手の負担に配慮しつつ、必要事項を明確に依頼する
- **確認・質問**: 回答しやすく論点を絞る
- **お詫び・謝意**: 誠実に、過剰にならない

殴り書きから明らかに 1 つに定まる場合は、この Step をスキップして Step 4 へ進んでよい。迷う場合は必ず聞く。

### Step 4: 清書

選ばれたトーン軸 + 固定のスタイル規範に沿ってベスト 1 案を作成する。複数案は出さない。

- 冒頭に結論（依頼・質問・報告の要点）
- 必要に応じて背景・前提を 1〜2 行
- 箇条書きは読みやすさが上がるときのみ使う（短文が並ぶだけなら地の文の方が良い）
- 末尾に「以上です」等の定型句は不要。ただし依頼系で返信アクションが必要なら、最後に簡潔に明示する
- **「お疲れ様です」で始めない** — 結論から入る

### Step 5: 出力

以下の形式で出力する。

1. コードブロック（```）で清書文のみを提示する（コピペしやすくするため）
2. コードブロックの外に補足や解説を書かない

例:

````markdown
```
〇〇の件、本日中にご確認お願いします。
...
```
````

清書結果以外の文言（トーン選択の理由、スタイル判断の根拠など）は出力しない。ノイズになる。
