# review

ベースブランチとの差分をレビュー指針に基づいて点検する。

## 使用方法

```sh
/my:review --base <ベースブランチ>
```

`--base` は必須。指定がなければ使い方を案内して即時終了する。

## 実行フロー

### Step 1: 差分収集

```bash
git diff <base>...HEAD --name-only
git diff <base>...HEAD
```

直近コミットのみとの比較は禁止。必ず `<base>...HEAD` を使う。

### Step 2: 差分の点検

変更されたファイルごとに、以下のレビュー観点すべてを適用して点検する。
必要に応じて変更されていないファイルも読み、文脈を把握する。

### Step 3: 結果報告

指摘がある場合のみ報告する。問題がなかったファイルには言及しない。

報告フォーマット:

```text
### <ファイルパス>

**L<行番号>**: <指摘内容>
<根拠（レビュー観点のどの条項に基づくか）>

<修正案があればコードブロックで提示>
```

指摘がゼロなら「指摘なし」とだけ報告する。

## レビュー観点

### 基本姿勢

- コードの美しさは、見た目の簡潔さではなく、責務、状態、分岐、依存関係が最小の規則で整理され、構造から仕様と変更影響が自然に読めることによって判断する
- 短くても曖昧なら不合格とし、宣言的で誤読しにくい構造を優先する
- コードを善意に解釈しない
- 表面的な可読性や一見の動作だけで合格にしない
- 問題がないことを探すのではなく、欠陥の可能性を潰す姿勢で点検する
- 以下の観点を前提に網羅的に確認する
  - 責務
  - 分岐
  - 状態管理
  - 副作用
  - 命名
  - 拡張性
  - テスト容易性
  - 重複
  - 境界条件

### 調査と根拠

- 判断に迷った場合は必ず Context7 で公式リファレンスを参照する
- 推測、慣習、二次情報で補完しない
- 一次情報を優先する
- 検索クエリと調査は必ず英語で行う
- 英語の公式ドキュメントを基準に理解と実装判断を行う
- 曖昧なまま進めず、根拠を確認してから進む

### 設計原則

#### 全体方針

- 後方互換性、既存仕様への追従、変更影響範囲、移行コスト、段階的改修のしやすさは考慮しない
- 現在の構造を守ることより、責務の再定義と全体最適化を優先する
- 不要なら壊してでも整理し、最も単純で宣言的かつ直線的に読める構造へ作り直す

#### 単一責任と構造

- 常に単一責任を優先する
- 関数・コンポーネントは短く保つ
- UI・状態管理・データ整形・副作用を混在させない
- `if/else` の多用や深いネストを避ける
- 早期 return と責務分割で直線的に読める構造を選ぶ
- 場当たり的な条件追加より構造改善を優先する
- 賢さより可読性を重視する

#### 宣言的設計と拡張性

- 条件追加のたびに `if` 連鎖が増える構造を避ける
- 条件テーブル、設定オブジェクト、マップなどで表現する
- 100件が1000件になっても破綻しないスケーラブルな構造を選ぶ

#### 命名とインターフェース

- 変数名・関数名・props名は役割と粒度を揃えて一貫させる
- 無意味な一時変数や説明のためだけの中間変数は排除する
- 命名と構造で意図を表現する
- props は責務ごとに整理し、場当たり的に増殖させない

#### コードコメント

- 原則禁止。コードを読めばわかることは書かない
- コメントが必要になった時点で、まず命名・構造・責務分割で意図を表現できないか見直す
- それでもコードだけでは背景や制約が伝わらない場合に限り、苦肉の策として書く
- 許容される例: 外部仕様の制約、非自明なワークアラウンドの理由、ビジネスルールの根拠
- 禁止される例: 処理の要約、セクション区切り、型や引数の説明、TODO 放置

#### TypeScript の型定義

- 組み込み Utility Types や mapped types を優先して宣言的に表現する
- 手書きの重複型や場当たり的な union 追加を避ける
- 型の責務と真実の所在を明確にし、変更に強く読みやすい構造を保つ
- 型は、その値が表す責務だけを表現し、取得中・未解決・欠損といった周辺都合を安易に混入させない
- あるコンポーネントや関数が「確定済みの値を扱う責務」を持つなら、その入力型は確定済みで完結させる
- `undefined` や `null` を型に含めるのは、その値の不在自体が仕様として意味を持つ場合に限る
- 本来仕様に存在しない `undefined` を型へ混ぜることで、下流に不要な防御分岐を発生させてはならない
- 「値の意味」と「値がまだ来ていない事情」を同じ型で表現しない。取得状態は取得状態として分離し、正常系には確定済みの値だけを渡す
- optional な型が本当に仕様上の意味を持つのか、上流の遅延や初期状態を押し付けているだけなのかを必ず判定する

#### SSOT

- 状態、条件、設定、表示根拠は必ず SSOT に集約する
- 同じ意味の値や判定を複数箇所に重複定義しない
- 派生値は都度計算または明示的に生成する
- コピーを持たない
- 真実の所在が一目で分かり、変更時に修正箇所が増えない構造を徹底する

### React コンポーネント設計

#### 責務の再評価

- 既存の React コンポーネントを責務単位で再評価する
- 責務逸脱・責務過多・存在意義の薄いものは分割、統合、削除する
- 場当たり的な延命ではなく、責務の再配置と構造改善で整理する

#### コンポーネント間の関係

- 親子、役割、責務境界、拡張点がコード上に自然に表れる設計を優先する
- compound pattern など、関係性が読み取れる構造を選ぶ
- props の寄せ集めで表現しない
- 関連部品を並べたときに、仕様と設計意図が同じ文法で読める一貫性を保つ

#### サイズと複雑さ

- 人間が一読で責務と仕様を把握できる大きさかを確認する
- 100行前後を目安とする
- 150行超は要分割、200行超は原則分割とする
- 行数だけでなく、分岐数、ネスト深さ、状態管理、副作用、表示責務の混在も厳しく見る
- 複雑なら子コンポーネントや custom Hook に抽出する

### useEffect の扱い

- `useEffect` を見つけたら、まず設計の歪み、責務混在、導出値の state 化、副作用の置き場所の誤りを疑う
- 存在を前提に正当化せず、本当に外部システムとの同期かを最初に確認する
- `useEffect` は React 公式が認める外部システムとの同期用途以外では使わない
- 描画用データの導出に使わない
- 状態の連鎖更新に使わない
- イベント処理の代替に使わない
- まず render 中の計算、event handler、`key`、props の見直し、custom Hook で解決できないか検討する

### useSWR 系 `data?.` の扱い

- `data?.` の多用は、取得状態と利用状態の分離不足から生まれる設計欠陥であり、責務分離の問題として扱う
- `data` を使う処理は `loading` / `error` を抜けた後に書く。`undefined` のまま optional chaining でつなぐ実装は禁止

#### 解決パターン

1. **Provider / Context による隔離**: 取得・`loading`・`error` を Provider 側で完結させ、下流には取得済みの値だけを渡す
2. **Content コンポーネントで状態を先に切る**: `error` → `loading` → empty → ready を先頭で分岐し、正常系ブロック内では `data` の存在を前提に書く

#### 禁止パターン

- `data?.xxx ?? []` で未取得と空データを同一視する
- `data?.xxx ?? false` で未取得と権限なしを同一視する
- 取得 hook を呼んだコンポーネント内で `data?.` のまま UI 分岐やイベント処理まで行う
- 「落ちないようにする」目的の optional chaining

### カスタムHook設計

#### Read/Write分離

- 一つのHookにデータ取得と変更操作を混在させない
- Read Hook（`useXxx`）はデータと取得状態を返す
- Write Hook（`useCreateXxx`, `useUpdateXxx`, `useDeleteXxx`）はアクション（関数）を返す
- Hookの戻り値がデータなのかアクションなのか、型と命名から一目でわかる状態を保つ

#### 一方向データフロー

- カスタムHookでもデータの流れは一方向に保つ
- Read Hookの結果をWrite Hookの入力に暗黙的に結合しない
- コンポーネントがRead HookとWrite Hookを個別に呼び、データフローをコンポーネント側で明示的に組み立てる

#### 変更後の再フェッチ

- create/update/delete後の再フェッチを各write Hookが個別に実行しない
- SWRの`mutate`をキーベースで使い、再検証のトリガーを集約する
- 複数の変更操作が同じデータの再フェッチを必要とする場合、Mediator的に一箇所で制御する
- write Hookは「変更の成功」だけを返し、再フェッチの責務を持たない

## 修正案を出す前の必須確認

修正案を提案する前に、必ず以下を実行する。

- 既存コードの設計意図を説明し、なぜそう書かれたかを述べる
- 修正案がレビュー観点のどの条項に基づくかを明示する
- 修正案自体がレビュー観点の他の条項に違反していないか検証する
- 技術判断を伴う場合は Context7 で公式ドキュメントを参照した根拠を示す
